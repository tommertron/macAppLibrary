# scripts

## `merge-community-prs.py`

One-shot drain of every open `[Community] …` PR.

### Why

Submissions come in through the in-app **Submit to Community** flow, which
opens a PR on this repo. The repo is set up so that the **source of truth is
the per-bundle directory** at `community-data/<bundleID>.json`, and the
monolithic `community-data.json` at the repo root is regenerated by CI on
every push to `main` (see `.github/workflows/regenerate-community-data.yml`).
The app fetches only the monolithic file.

PRs can land in three shapes, all of which this script handles:

| Shape       | What the PR contains                              | What the script does                                                                 |
| ----------- | ------------------------------------------------- | ------------------------------------------------------------------------------------ |
| **Legacy**  | edit to monolithic `community-data.json`          | extract the new entry, write `community-data/<bundle>.json`, push, close PR          |
| **Add**     | new `community-data/<bundle>.json`                | `gh pr merge --squash --delete-branch` with rebase-retry                             |
| **Update**  | modified existing `community-data/<bundle>.json`  | skipped by default (data change); pass `--include-updates` to merge them too         |

Anything that doesn't fit those shapes is listed as `unknown` and left alone.

### Usage

```sh
scripts/merge-community-prs.py                    # legacy + Adds (default)
scripts/merge-community-prs.py --include-updates  # everything, including data updates
scripts/merge-community-prs.py --dry-run          # preview without doing anything
```

Requires `gh` (authenticated) and `git` on PATH. Anchors to the repo root, so
it can be run from anywhere.

### Output

Default run starts with a summary:

```
Open community PRs: 16
  legacy:  0
  add:     15
  update:  1  (will skip — pass --include-updates to merge)
  unknown: 0
```

Each section then reports per-PR progress with `✓` / `✗`. If Updates were
skipped, their diffs are printed at the end so you can decide whether to
re-run with `--include-updates` or merge them by hand.

### Race handling

During a bulk-merge spree GitHub will occasionally reject a merge with "Base
branch was modified" because the regenerate workflow advanced `main` in
between. The script retries each merge up to 6 times with a short sleep, and
pushes from the legacy-handling step also rebase-and-retry on rejection. The
regenerate workflow itself does the same (`.github/workflows/regenerate-community-data.yml`),
so the system self-heals on contention rather than failing.

### When NOT to use it

- If a PR contains edits beyond a single per-bundle file or a single legacy
  edit, it's flagged `unknown` and skipped. Inspect those by hand
  (`gh pr diff <num>`).
- If you've staged unrelated work on `main` locally, commit or stash it first
  — the legacy path makes commits directly on the current branch.
